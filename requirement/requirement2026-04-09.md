# 需求说明（2026-04-09）

## 一、项目整体目标

`QWire` 是一套 AI 辅助开发的自动化测试平台，用于构建可扩展的测试能力，覆盖：

- API 测试（第一阶段）
- Web 自动化测试（第二阶段）
- App 自动化测试（第三阶段）

项目采用 AI 驱动开发模式（AI-assisted development），通过明确的 `Plan -> Schema -> Spec -> Code` 流程，确保代码生成的可控性、可维护性和一致性。

---

## 二、第一阶段项目结构

整个 workspace 包含 3 个工程模块：

### 1. `QWireMockAPI`：模拟服务设计与开发

**目标**：提供稳定、可控的测试依赖环境。

**主要工作包括：**

1. 定义模拟服务能力（接口、行为、状态）
2. 设计模拟服务结构（路由、状态管理、数据驱动）
3. 实现模拟服务（API、回调、状态机）
4. 支持测试场景（异常、延迟、错误注入等）

### 2. `QWireCore`：自动化测试框架设计与开发

**目标**：构建可扩展、可复用的测试执行框架。

**主要工作包括：**

1. 整理需求（测试能力、执行模型、扩展机制）
2. 设计框架（模块划分、执行流程、插件机制）
3. 建立规范（Schema、Spec、DSL、Checker 设计）
4. 生成样例（测试用例定义、数据结构、典型场景）
5. 生成代码（核心框架实现）
6. 生成测试用例（静态验证框架能力）

### 3. `QWireTestAPI`：测试工程设计与开发

**目标**：构建具体的测试项目与用例体系。

**主要工作包括：**

1. 整理测试需求（覆盖范围、场景划分）
2. 设计测试结构（目录、数据组织、用例划分）
3. 建立测试规范（命名规则、数据结构、断言方式）
4. 生成测试用例（单用例、多用例、组合场景）
5. 开发测试代码（执行逻辑、DataProvider、集成）

## 三、产出物要求

1. 建立本项目的总体 `plan` 文件；每个工程下可设置模块，且工程和模块都应有各自的 `plan` 文件。
2. 规划好 `blueprint` 下的目录结构：
   - 对于不划分模块的工程，在工程目录下建立 `blueprint/` 目录，以及记录 `schema/`、`spec/` 的子目录。
   - 对于划分模块的工程，在模块目录下建立 `blueprint/` 目录，以及记录 `schema/`、`spec/` 的子目录。
