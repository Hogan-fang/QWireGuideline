# Project Guideline

> 本文档仅保留 **技术性约定**。
> 与需求归类、变更审批、AI 行为边界、方法流程相关的内容，请分别参考 `AI-guideline.md` 与 `methodology.md`，此处不再重复。

## 1. 命名约定（Naming）

- 工程中的 API 请求/响应字段使用 `camelCase`
- 数据库表名与字段使用 `snake_case`
- Python 工程命名采用 `snake_case`
- 目录命名采用 `snake_case`
- 工程目录名与 Git 仓库名保持一致
- 枚举值（状态等）使用全大写字符串，如 `SUCCESS`、`FAIL`

## 2. 数据转换约定

- 跨层命名差异必须通过显式转换处理，并统一使用公共转换函数或工具（如 `failReason` ↔ `fail_reason`）
- 对于复杂对象或字段较多的转换场景，应使用映射结构或转换工具统一处理，避免大段重复赋值代码

## 3. 数据与安全（Data & Security）

- 敏感字段必须进行脱敏处理（如部分显示或掩码）
- 严禁在响应中返回敏感原始数据
- 所有接口数据必须符合对应的 OpenAPI Schema 定义

## 4. 错误处理（Error Handling）

- 参数校验失败、资源不存在、请求方法不支持或系统异常等 **HTTP / 框架层问题**，必须使用 HTTP 错误响应（如 `400`、`404`、`405`、`422`、`500`），并返回统一结构：

```json
{ "code": "error_code", "detail": "error description" }
```

- 业务处理失败时，除非有明确例外约定，否则应返回 **正常业务响应结构**，并通过业务状态字段（如 `status=FAIL`）及业务错误信息字段（如 `failReason`）表达失败原因
- API 设计中必须明确区分：**成功响应结构**、**业务失败响应结构**、**HTTP 错误响应结构**
- 成功响应中，不得混入 HTTP 错误结构字段（如 `code`、`detail`）
- HTTP 错误响应中，不得混入业务结构字段（如 `status`、`failReason`）
- HTTP 状态码必须使用标准语义，下表是必须实现和遵守的最小状态码语义集合

| 状态码 | 名称 | 含义 | 常见场景 |
| :---: | --- | --- | --- |
| `400` | Bad Request | 请求参数错误或格式不合法 | JSON 格式错误、缺字段、数据校验错误 |
| `404` | Not Found | 资源不存在 | 查询不到订单 |
| `405` | Method Not Allowed | 请求方法不支持 | 使用 `GET` 调用 `POST` 接口 |
| `409` | Conflict | 请求与当前资源状态冲突，无法处理 | 重复提交、幂等校验冲突、资源状态冲突 |
| `422` | Unprocessable Entity | 参数语法正确但语义错误 | UUID 格式错误、字段值非法 |
| `500` | Internal Server Error | 系统异常 | 空指针、系统崩溃等非受控故障 |



## 5. 日志（Log）

- 日志中的数据应记录为可读的 JSON 结构

## 6. 时间与时区

- 所有时间字段统一使用 ISO 8601 格式
- 所有时间统一使用 UTC 时间
- 数据库中的时间字段必须以 UTC 时间存储
- 禁止返回无时区语义的时间字符串
